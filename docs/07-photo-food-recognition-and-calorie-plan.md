# 拍照识别食物与热量估算方案

## 结论

Light Flow 可以做拍照识别和热量估算，但不应该宣传成“拍一张照片就准确计算热量”。

更可靠的定位是：

> AI 负责第一眼识别，Light Flow 负责营养计算，用户负责最后校正。

也就是说，拍照识别应该做成“可修正的估算流程”，而不是假装完全精准。

## 为什么不能只靠照片精准算热量

单张食物照片存在天然限制：

- 看不出真实重量。
- 看不出用了多少油。
- 看不出酱汁、糖、盐、奶油等隐藏配料。
- 混合菜很难拆分每种食材比例。
- 外卖、盖饭、沙拉碗经常有遮挡。
- 同一道菜不同做法热量差异很大。

因此任何模型都只能给出估算值。要让结果可信，必须允许用户改食物和份量。

## GitHub 和开源项目情况

GitHub 上有相关项目和数据集，但大多是研究型项目，不能直接搬进 Light Flow 当成完整功能。

### Nutrition5k

地址：https://github.com/google-research-datasets/Nutrition5k

特点：

- Google 的食物营养数据集。
- 包含餐盘、食材重量、热量、蛋白质、脂肪、碳水等信息。
- 适合参考数据结构和研究方向。

不适合直接使用的原因：

- 数据和流程偏研究。
- 项目已归档。
- 不能直接变成轻量 Web App 的拍照识别功能。

### V-Nutri

地址：https://github.com/K624-YCK/V-Nutri

特点：

- 研究用项目。
- 使用做菜过程视频和最终菜品图估算营养。
- 目标是估算热量、蛋白质、脂肪和碳水。

不适合直接使用的原因：

- 输入依赖做菜过程视频，不只是单张餐食照片。
- 模型和流程比较重。
- 更像研究验证，不适合直接嵌入当前 Light Flow。

### FoodSeg103

特点：

- 偏食物分割。
- 可以帮助判断图片里哪些区域属于某类食物。

不适合直接使用的原因：

- 它解决的是“图里有什么食物区域”，不是“这顿饭多少热量”。
- 分割之后仍然需要估算重量和营养。
- 对 Light Flow 当前阶段来说过重。

## 不建议的做法

- 不建议直接照搬 GitHub 上的研究代码。
- 不建议自己训练完整食物识别模型。
- 不建议把识别结果说成精准值。
- 不建议只返回一个总热量，不展示食物拆分。
- 不建议没有用户校正就直接写入记录。

## 推荐方案：Light Flow 拍照识别 MVP

第一版应该做成轻量、可信、可校正的流程。

流程：

1. 用户上传或拍摄餐食照片。
2. 页面展示照片预览。
3. 后端调用视觉模型识别可能的食物。
4. 模型返回食物名称、估算克数、置信度和说明。
5. Light Flow 根据自己的营养表计算热量和三大营养素。
6. 用户可以增删食物。
7. 用户可以调整每项食物份量。
8. 页面重新计算总热量、蛋白质、碳水、脂肪。
9. 用户确认后写入今日记录。

## 技术方向

### 1. 视觉识别

可以使用支持图片输入的视觉模型。

输入：

- 用户上传的餐食照片。
- Light Flow 支持的食材列表。
- 要求模型只从支持食材中匹配，无法确认时返回 `unknown`。

输出：

- 识别到的食物。
- 对应 Light Flow 食材 ID。
- 估算克数。
- 置信度。
- 是否需要用户确认。
- 简短说明。

### 2. 营养计算

热量不要完全由模型自由生成。

更可靠的方式是：

- 模型估算食物和份量。
- Light Flow 使用本地营养表计算热量、蛋白质、碳水和脂肪。

这样结果更稳定，也更容易调试。

### 3. 用户校正

每个识别结果都应该能修改：

- 食物名称
- 食材匹配
- 克数或份量
- 是否保留该项

用户确认后再写入每日记录。

## 推荐返回数据结构

```json
{
  "items": [
    {
      "name": "鸡胸肉",
      "matchedIngredientId": "chicken-breast",
      "estimatedGrams": 120,
      "confidence": 0.78,
      "needsReview": false,
      "note": "估算为一小块熟鸡胸肉"
    },
    {
      "name": "米饭",
      "matchedIngredientId": "white-rice",
      "estimatedGrams": 150,
      "confidence": 0.72,
      "needsReview": true,
      "note": "份量可能受餐盘角度影响，需要确认"
    }
  ],
  "summary": {
    "needsReview": true,
    "reason": "部分食物份量只能估算"
  }
}
```

营养汇总由 Light Flow 根据 `matchedIngredientId` 和 `estimatedGrams` 计算。

## 前端交互建议

拍照页可以分为四个状态：

### 空状态

- 上传餐食照片。
- 提示支持餐盘、便当、外卖盒。
- 提醒结果可以手动调整。

### 识别中

- 展示照片预览。
- 显示“正在识别食物和估算份量”。

### 待确认

- 展示每个识别项。
- 每项显示食物名、估算份量、热量、置信度。
- 支持加减份量。
- 支持删除识别项。
- 支持新增食物。

### 已记录

- 显示已写入今日记录。
- 可跳转到今日记录页。

## 后端实现建议

如果部署在 Netlify，可以使用 Netlify Function：

```text
netlify/functions/scan-meal.js
```

职责：

- 接收前端上传的图片。
- 调用视觉模型。
- 返回结构化识别结果。
- 不在前端暴露 API Key。

前端只负责：

- 上传图片。
- 展示识别结果。
- 用户校正。
- 调用本地营养计算。
- 写入每日记录。

## 与 Light Flow 的结合方式

拍照识别不应该是孤立功能，而应该和当前产品核心能力结合：

- 识别出的食物可以匹配 Light Flow 已有食材 ID。
- 识别出的食物可以写入今日记录。
- 如果识别到用户冰箱里没有的食材，可以提示“加入常备食材”。
- 如果识别结果和某道菜谱高度相似，可以推荐对应菜谱。
- 长期记录可以帮助用户看到每日热量和蛋白质趋势。

## 第一版验收标准

第一版不要求完全准确，但要可信、可改、能记录。

验收标准：

- 可以上传照片并显示预览。
- 可以返回至少 1-5 个可能食物项。
- 每个食物项都有名称、估算份量和置信度。
- 用户可以修改份量。
- 用户可以删除错误识别项。
- 用户可以新增漏识别食物。
- 页面能根据修正后的份量重新计算营养。
- 用户确认后能写入今日记录。
- 页面文案明确这是估算值，不是假装精准值。

## 推荐执行顺序

建议顺序：

1. 先补一份 Light Flow 食材营养表。
2. 把拍照页从纯上传入口改成四状态流程。
3. 先用模拟识别结果打通前端体验。
4. 再接入后端视觉识别 API。
5. 最后根据真实测试照片优化提示词和营养计算。

## 最终判断

Light Flow 可以写自己的拍照识别热量估算功能。

但正确方向不是照搬 GitHub 研究项目，也不是承诺绝对准确，而是做成：

> 上传照片 → AI 估算食物和份量 → Light Flow 计算营养 → 用户校正 → 写入记录。

这样最符合 Light Flow 的轻食助手定位，也最容易逐步上线和验证。
