# Grocy 与 Light Flow 结合使用判断

## 结论

Grocy 可以和 Light Flow 结合使用，但不建议直接把 Grocy 嵌进 Light Flow，也不建议照搬 Grocy 的产品形态。

更合适的定位是：

> Grocy 是“冰箱数据库”，Light Flow 是“今天吃什么的轻食助手”。

代码层面也不应该完全照搬 Grocy。Grocy 可以作为能力参考和可选外部数据源，但具体实现需要和 Light Flow 现有的数据结构、页面体验、轻食推荐逻辑结合起来。

也就是说：

- Grocy 负责重型家庭库存能力。
- Light Flow 负责移动端轻食推荐、摄入记录、拍照识别和轻量决策。

## Grocy 适合承担什么

Grocy 是一个开源的家庭库存管理系统，适合管理：

- 家庭食材库存
- 保质期和临期提醒
- 购物清单
- 条码扫描
- 家庭用品消耗
- 菜谱和 Meal Plan
- 库存相关 API

它更像家庭冰箱和家庭物资的 ERP，能力比较重，适合做底层库存系统。

## Light Flow 应该继续承担什么

Light Flow 不应该变成另一个 Grocy。

Light Flow 更应该继续聚焦：

- 移动端优先的轻食体验
- 根据已有食材推荐能做的菜
- 控制热量、蛋白质和每日摄入
- 拍照识别这一餐
- 菜谱图片和食谱内容一致
- 更轻、更快、更适合普通用户做饮食决策

## 推荐结合方式

### 1. 从 Grocy 同步库存

从 Grocy 读取用户家里的库存食材，再映射到 Light Flow 的食材 ID。

示例：

- Grocy 里有鸡蛋、番茄、西兰花。
- Light Flow 自动把这些食材设为已拥有。
- 首页和发现页根据这些食材推荐当前可做菜谱。

这样可以减少用户在 Light Flow 里重复维护食材库存。

### 2. 用保质期影响推荐排序

Grocy 的强项是库存和保质期。

Light Flow 可以读取临期食材，然后提高相关菜谱的推荐优先级。

示例：

- 菠菜明天过期。
- Light Flow 优先推荐菠菜鸡蛋、菠菜豆腐汤、菠菜蘑菇炒蛋。

这个方向和 Light Flow 的“冰箱轻食师”定位非常匹配。

### 3. 把缺少食材写入 Grocy 购物清单

当 Light Flow 推荐一道菜，但用户缺少部分食材时，可以提供“加入购物清单”的动作。

示例：

- 用户想做三文鱼牛油果生菜碗。
- 当前库存缺三文鱼和牛油果。
- Light Flow 可以把缺少的食材写回 Grocy 购物清单。

这样 Grocy 继续负责购物清单，Light Flow 不需要重复做一套复杂库存系统。

### 4. 不用 Grocy 管 Light Flow 的菜谱图片

当前 Light Flow 的核心问题之一是菜谱图片和食谱不对应。

Grocy 的强项不是高质量轻食图片库，所以不建议把菜谱图片交给 Grocy 管。

菜谱图片、菜名、食材和步骤仍然应该由 Light Flow 自己维护。

### 5. 不照搬 Grocy 的界面和信息架构

Grocy 是重型家庭库存工具，信息密度高，功能很多。

Light Flow 是轻食助手，应该保持轻量、移动端优先、决策路径短。

如果直接照搬 Grocy，Light Flow 会变成复杂的家庭 ERP，反而丢掉当前产品最有差异化的轻体验。

### 6. 不直接照搬 Grocy 代码

即使未来参考 Grocy，也不应该把它的代码整块复制到 Light Flow。

原因：

- Grocy 的架构服务于自托管库存系统，和 Light Flow 当前的轻量前端结构不同。
- Grocy 的数据模型偏库存、采购、消耗和保质期，Light Flow 的数据模型偏食材、菜谱、营养和推荐。
- 直接搬代码会让项目变重，也会增加后续维护难度。
- Light Flow 当前最重要的是移动端轻食体验，不是完整家庭库存后台。

更合适的做法是“能力转译”：

- 参考 Grocy 的库存、保质期、购物清单能力。
- 把这些能力改写成适合 Light Flow 的轻量版本。
- 只保留对 Light Flow 推荐逻辑有帮助的数据。
- 页面上仍然使用 Light Flow 自己的交互和视觉风格。

例如，不是复制 Grocy 的库存页面，而是在 Light Flow 的食材页增加更轻的能力：

- 已有食材
- 数量
- 快过期提醒
- 一键优先推荐
- 缺少食材加入购物清单

## 建议产品关系

推荐关系：

```text
Grocy
  ↓ 提供库存、保质期、购物清单
Light Flow
  ↓ 生成轻食推荐、摄入记录、拍照识别结果
用户
```

也可以理解为：

- Grocy 回答“家里有什么、什么时候过期、还缺什么”。
- Light Flow 回答“今天应该吃什么、怎么做、吃了多少热量和蛋白质”。

## 不建议现在立即做的原因

当前 Light Flow 更紧急的问题是：

1. 菜谱图片和食谱不对应。
2. 菜谱数量和质量还需要扩充。
3. 拍照识别还没有形成完整可修正流程。

Grocy 集成可以增强库存来源，但不能自动解决以上三个核心体验问题。

如果现在优先接 Grocy，可能会让系统能力变重，但用户看到的菜谱图片、内容数量和拍照体验仍然没有明显改善。

## 推荐执行顺序

建议按以下顺序推进：

1. 先修复菜谱图片和食谱对应问题。
2. 把菜谱扩充到 90-100 道，并保证图文一致。
3. 把拍照识别做成可修正、可写入记录的流程。
4. 再考虑 Grocy 集成。

## 未来 Grocy 集成最小版本

如果后续要做 Grocy 集成，建议先做最小版本，不要一次做全量功能。

最小版本可以包括：

- 配置 Grocy API 地址和 API Key。
- 从 Grocy 拉取库存列表。
- 建立 Grocy 商品名到 Light Flow 食材 ID 的映射。
- 自动设置 Light Flow 已有食材。
- 根据临期食材提高菜谱推荐优先级。

暂时不做：

- 不同步 Grocy 菜谱。
- 不同步 Grocy 菜谱图片。
- 不复制 Grocy 的库存管理界面。
- 不在 Light Flow 里重做完整家庭库存系统。
- 不整块搬 Grocy 代码到 Light Flow。
- 不让 Grocy 的重型信息架构覆盖 Light Flow 的轻食体验。

## Light Flow 化改造原则

未来如果参考 Grocy，需要遵守以下原则：

- 先看 Light Flow 当前用户需要什么，再决定引入 Grocy 的哪一部分能力。
- 只引入能帮助“今天吃什么”的库存信息。
- 任何库存能力都要服务于菜谱推荐、临期食材优先和饮食记录。
- UI 不照搬 Grocy，继续保持 Light Flow 的移动端轻量体验。
- 数据结构要适配 Light Flow 现有食材 ID 和菜谱 ID。
- 代码实现要局部、可维护，不为了接 Grocy 重构全站。
- 如果接 API，只做清晰的同步层，不让外部系统侵入核心菜谱逻辑。

## 最终判断

Grocy 适合作为 Light Flow 的外部库存能力来源，但不适合作为 Light Flow 的产品主体。

Light Flow 应该保持自己的核心体验：

- 看图就知道吃什么。
- 选食材就能推荐菜。
- 拍照后能记录这一餐。
- 饮食决策轻、快、可信。

Grocy 的价值在后面，不在当前第一优先级。
