# 第一阶段路线图：内容可信度修复

## 1. 阶段目标

第一阶段的目标是先让用户看到的内容可信。

优先解决：

- 菜谱图片和食谱不对应。
- 菜谱数量偏少。
- 菜谱步骤质量不够稳定。
- 食材 ID 和菜谱引用不完全一致。

暂时不优先：

- React/Vite 迁移。
- Grocy 集成。
- 真实 AI 拍照识别。
- 大规模重做全站样式。

## 2. 为什么先做内容可信度

Light Flow 当前最大体验风险是“用户一眼看到菜谱图和菜名不一致”。

如果这个问题不先修复，即使后面接入拍照识别、库存同步或 React 重构，用户仍然会觉得产品不可信。

第一阶段应该先把当前可见内容修稳，再做更大的技术升级。

## 3. 阶段任务

### 任务 1：确认 GitHub 和预览工作流

目标：

- 确认当前项目是否已经关联 GitHub。
- 确认 Netlify 是否连接 GitHub。
- 确认 Pull Request 是否能生成 Deploy Preview。

交付：

- `docs/modules/deploy-preview.md` 更新为真实状态。
- 如果尚未连接 GitHub，记录需要用户确认的步骤。

### 任务 2：修复菜谱图片匹配

依据：

```text
docs/11-recipe-image-audit.md
```

目标：

- 修复所有“不合格”菜谱图片。
- 保留或标记“可接受近似”。
- 不新增菜谱。
- 不改拍照识别。

交付：

- 修改后的 `app.js`。
- 更新后的菜谱图片审计表。
- 本地验证结果。
- 预览链接。

### 任务 3：统一食材数据

目标：

- 处理菜谱引用但顶部食材列表缺失的 ID。
- 确认这些食材是否有图标、单位和分类。
- 避免菜谱匹配和食材展示出现缺失。

交付：

- 食材数据补齐或重整方案。
- 必要时更新 `app.js`。
- 验证所有菜谱引用食材 ID 都存在。

### 任务 4：扩充菜谱到 90-100 道

目标：

- 从已有图库反推新菜谱。
- 保证新增菜谱图文一致。
- 覆盖早餐、午餐、晚餐、加餐。
- 避免只堆少数食材。

建议结构：

```text
早餐：20 道
午餐：35 道
晚餐：35 道
加餐：10 道
```

交付：

- 新增菜谱数据。
- 新增或更新菜谱详情。
- 图片匹配审计同步更新。

### 任务 5：提升菜谱步骤质量

目标：

- 每道菜在展示层保持 6 步。
- 重点菜谱补充更具体的备料、处理、烹饪、调味和出餐判断。
- 避免空泛步骤。

交付：

- 更新后的菜谱步骤。
- 发现页和详情页复查。

## 4. 第一阶段不做什么

第一阶段不做：

- 不迁移 React/Vite。
- 不接真实 AI 视觉识别。
- 不接 Grocy。
- 不重新设计全站。
- 不引入大型 UI 组件库。
- 不直接改正式站，优先走预览。

## 5. 验收标准

第一阶段完成时应满足：

- 所有当前菜谱图片没有明显错配。
- 菜谱数量达到 90-100 道。
- 新增菜谱都使用已有食材 ID。
- 所有菜谱引用的食材 ID 都可被识别。
- 发现页和菜谱详情页图片加载正常。
- 移动端页面没有明显文字溢出。
- `node --check app.js` 通过。
- 有 Netlify 预览链接供确认。

## 6. 推荐分支

建议分支：

```text
feature/phase-1-content-trust
```

如果要更细，可以拆成：

```text
feature/recipe-image-fix
feature/ingredient-data-unification
feature/recipe-expansion-100
```

## 7. 推荐执行顺序

```text
确认 GitHub/Netlify 预览
  ↓
修复现有菜谱图文不一致
  ↓
统一食材数据
  ↓
基于图库扩充菜谱
  ↓
补强步骤质量
  ↓
预览确认
  ↓
合并正式站
```
