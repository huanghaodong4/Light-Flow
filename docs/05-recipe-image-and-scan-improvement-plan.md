# 菜谱图片、菜谱扩充与拍照识别改进方案

## 核心问题

当前项目最需要解决的不是单点功能问题，而是三个用户信任问题叠在一起：

1. 菜谱和图片不匹配，用户第一眼就会不信。
2. 菜谱数量不够，且扩充方式还没有稳定体系。
3. 拍照识别更像占位功能，还没有形成可校正、可记录的完整流程。

这三个问题会互相放大：如果菜谱图片不可信，即使拍照识别入口做得更强，最终推荐出来的结果也仍然会显得不专业。

## 问题 1：菜谱和图片不匹配

### 当前表现

项目里存在明显的借图现象，例如：

- 豆腐煲使用了麻婆豆腐图片。
- 牛肉蘑菇糙米饭使用了牛肉面图片。
- 虾仁黄瓜炒蛋使用了其他轻食碗图片。

这类问题会直接影响用户对产品的信任。用户看到菜名和图片不一致时，很容易判断整个菜谱推荐不可靠。

### 解决方向

先建立菜谱图片审计表，把每道菜分为三类：

- 精确匹配：图片就是这道菜，或几乎完全一致。
- 可接受近似：主要食材、形态、餐盘类型基本一致。
- 不合格：菜名、主要食材或菜品形态明显不对应。

处理原则：

- 不合格图片必须替换。
- 没有合适图片的菜谱，优先改菜名或暂时下架，不要继续硬配图。
- 后续新增菜谱应优先从已有可用图片反推菜谱，而不是先写菜谱再找图。

### 能解决到什么程度

这份方案本身只能解决“怎么判断、怎么处理、按什么顺序做”的问题，还不能直接解决“图片和食谱已经对应准确”的结果问题。

真正把图片和食谱对应问题解决好，需要继续完成一轮落地执行：

1. 逐道菜审计当前所有菜谱。
2. 对照菜名、主食材、菜品形态和当前图片。
3. 给每道菜标记匹配等级。
4. 修掉所有不合格图片。
5. 对没有合适图片的菜谱，改菜名、换菜谱或暂时下架。
6. 修改后在发现页和菜谱详情页复查展示效果。

建议新增一个审计文件，例如：

```text
docs/06-recipe-image-audit.md
```

审计表建议包含以下字段：

- 菜谱 ID
- 当前菜名
- 当前图片路径
- 匹配等级
- 问题说明
- 处理方式
- 修复后图片路径或修复后菜名

只有完成这张审计表，并根据审计结果修改 `app.js` 里的菜谱数据，图片和食谱对应问题才算真正明显改善。

## 问题 2：菜谱数量和质量不足

### 当前表现

用户感知中当前只有七十几道菜谱，但当前代码里的 `recipe()` 条目数量并不多，且一部分菜谱依赖近似图片。

图库报告显示，项目已有较多可用图片素材，说明现在的主要瓶颈不是完全没有图片，而是还没有把图库、食材、菜谱名、营养、步骤稳定绑定起来。

### 解决方向

建议先把菜谱扩充到 90-100 道，并保证每道菜都满足基础质量标准。

推荐结构：

- 早餐：20 道
- 午餐：35 道
- 晚餐：35 道
- 加餐：10 道

每道菜谱应包含：

- 菜谱 ID
- 中文菜名
- 已有食材 ID
- 匹配图片
- 热量
- 蛋白质
- 制作时间
- 评分
- 简短描述
- 餐次类型
- 食材份量
- 至少 6 步跟做说明

扩充原则：

- 只使用已有食材 ID。
- 优先使用已有高质量图片。
- 图片必须和菜名、主食材、菜品形态一致。
- 不要只堆鸡胸肉、西兰花、鸡蛋等少数食材。
- 覆盖早餐、午餐、晚餐、加餐。
- 覆盖冷食碗、热炒、汤、吐司、燕麦杯、饭碗、意面、沙拉等多种类型。

## 问题 3：拍照识别水平不够

### 当前表现

当前“拍照识别这一餐”页面主要是上传入口和提示文案，还没有真正形成以下流程：

- 上传或拍摄照片
- 展示照片预览
- 给出识别结果
- 拆分食物和份量
- 估算热量和营养
- 允许用户手动修正
- 写入每日记录

因此用户会感觉拍照功能整体水平不高。问题并不只是识别模型准不准，而是整个识别结果的产品闭环还没有建立。

### 解决方向

拍照识别应先做成“可修正的可信流程”。

第一阶段可以不直接接真实 AI 识别，而是先完成体验闭环：

1. 用户上传照片。
2. 页面显示照片预览。
3. 系统给出模拟识别结果。
4. 用户可以增删食物。
5. 用户可以调整份量。
6. 页面计算热量、蛋白质、碳水、脂肪。
7. 用户确认后写入今日记录。

第二阶段再接真实视觉识别模型。

即使接入真实 AI，也必须保留手动调整，因为餐食识别天然会有误差，尤其是混合菜、外卖、酱汁、隐藏食材和份量估算。

## 推荐执行顺序

### 第一阶段：修复菜谱图片匹配

目标：先恢复用户对菜谱内容的信任。

工作内容：

- 审计当前全部菜谱图片。
- 标记精确匹配、可接受近似、不合格。
- 替换明显不对应图片。
- 对没有合适图片的菜谱，改名或暂时移除。
- 保证发现页和菜谱详情页展示的图片都可信。

预计影响范围：

- `app.js`：菜谱图片路径、菜名、描述。
- `lightflow-food-image-database/`：只读取图库素材，不随意删除或替换原图。
- `verification/`：可新增审计截图或审计结果。

### 第二阶段：扩充菜谱到 90-100 道

目标：让用户感觉内容明显更丰富，同时保持图文一致。

工作内容：

- 从已有图库反推新菜谱。
- 补充早餐、午餐、晚餐、加餐覆盖。
- 每道新增菜谱使用已有食材 ID。
- 每道菜谱补齐营养、时间、描述、餐次、份量和 6 步说明。
- 避免重复菜名和高度相似组合。

预计影响范围：

- `app.js`：新增菜谱数据和菜谱详情数据。
- 不重构代码结构。
- 不改全站设计。

### 第三阶段：升级拍照识别流程

目标：让拍照入口从占位页变成可用的记录流程。

工作内容：

- 增加图片上传后的预览状态。
- 增加识别结果列表。
- 支持手动增删食物。
- 支持调整份量。
- 估算营养数据。
- 确认后写入今日记录。

预计影响范围：

- `app.js`：拍照页状态、识别结果、记录写入逻辑。
- `styles.css`：拍照页局部样式。
- 不改其他页面结构。

## 不建议的做法

- 不建议先大改拍照识别，再回头修菜谱图片。
- 不建议继续用明显不对应的图片撑菜谱数量。
- 不建议为了扩数量而添加没有图片、没有步骤、营养不合理的菜谱。
- 不建议第一轮就重构 `app.js`。
- 不建议重新设计全站。

## 建议优先任务

下一步最适合先做：

1. 生成当前菜谱图片审计表。
2. 修复所有不合格图片匹配。
3. 基于已有图库扩充到 90-100 道菜谱。
4. 补齐全部菜谱的 6 步跟做说明。
5. 最后再升级拍照识别流程。

这个顺序能先解决最影响观感和信任的问题，也能为后续 AI 拍照识别打好内容基础。

## 下一步落地标准

如果下一轮正式开始修复，第一步不应该直接批量新增菜谱，而应该先产出 `docs/06-recipe-image-audit.md`。

验收标准：

- 当前所有菜谱都出现在审计表里。
- 每道菜都有明确的图片匹配等级。
- 所有“不合格”项都有处理方案。
- 修改后的 `app.js` 不再保留明显菜名和图片不对应的菜谱。
- 新增菜谱必须从已有可用图片反推，避免继续制造新的图文不一致问题。
